Claire O'Brien (Tech Lead)                                               10:07
  Before we go too deep into scope -- I want to flag some technical
  considerations early so we don't paint ourselves into a corner. The current
  booking system has a fairly monolithic architecture. Seat availability and
  inventory is managed by the PSS -- our Passenger Service System -- which is
  the Amadeus Altea integration. Any real-time seat map or upgrade
  recommendation logic will need to call into that system. The challenge is
  that Altea API calls have a latency ceiling we need to respect, especially if
  we want to surface personalised recommendations mid-flow without degrading
  the booking experience.

James Nguyen (Business Analyst)                                          10:09
  So are you saying we can't do real-time seat recommendations? Or just that we
  need to be careful about when we call the API?

Claire O'Brien (Tech Lead)                                               10:10
  We can do real-time, but we need to think about caching strategies and where
  in the funnel we trigger the calls. What I'd suggest is that we fetch seat
  availability once when the passenger selects their flight, and cache that
  inventory snapshot for the duration of that session -- maybe 15 minutes --
  before refreshing. We should avoid calling Altea on every page render. That
  should give us acceptable performance. The recommendation logic itself -- the
  "you might like" personalisation -- could be a separate microservice we build
  in-house using booking history and loyalty tier data.

Sarah Mitchell (Product Owner)                                           10:12
